## 项目简介

这个项目不是 Claude Code 的复刻代码，而是一套源码级架构分析。它从庞大代码库中提炼 Agent Loop、上下文工程、工具、记忆、Skills、MCP、多 Agent、权限和用户体验等主题，既解释“它怎么实现”，也追问“为什么要这样设计”。

项目的价值在于提供一张生产级 Coding Agent 地图：不只展示主流程，还覆盖并发、故障恢复、Token 预算、权限防御和多 Agent 隔离等容易被最小 Demo 忽略的问题。

## 为什么值得学习

- 它研究的是经过真实复杂任务检验的系统，而不是只跑通理想路径的 Demo。
- 它把约 51.2 万行源码压缩为 15 个可独立阅读的专题，降低进入门槛。
- 它同时关注模型能力与工程约束：速度、安全、成本、恢复和可观测性。
- 它能帮助我把 Claude Code 的产品体验对应到底层设计，而不只模仿界面。
- 它和 Claude Code From Scratch 互补：先看生产系统的完整取舍，再动手理解最小实现。

## Claude Code 的整体架构

Claude Code 可以看成围绕 QueryEngine 展开的会话与执行系统：

1. 用户输入进入 QueryEngine，由 query 主循环组织本轮任务。
2. 主循环调用 Claude API，并持续解析流式响应。
3. 文本内容直接进入终端输出；工具调用进入统一执行引擎。
4. 工具结果回注消息历史，成为下一轮模型判断的新观察。
5. System Prompt、Git 状态、CLAUDE.md 和压缩结果共同构成上下文。
6. 所有工具执行都先经过权限规则、Bash AST 分析和必要的用户确认。

真正值得学习的不是某一个模块，而是这些模块如何围绕同一条主循环协作：上下文给模型判断依据，工具改变环境，权限控制风险，压缩和记忆保证任务可以继续。

## Agent Loop

Claude Code 的主循环不是简单的“请求模型—调用工具—重复”。项目重点分析了：

- 双层循环：外层管理会话与生命周期，内层处理模型响应和工具执行。
- 全链路流式：模型输出生成时就开始解析和展示。
- 工具预执行：工具参数还在流式生成时便提前准备或启动，隐藏部分等待时间。
- 并发控制：可安全并行的只读工具并发，写操作保持串行。
- 7 类 Continue Sites：针对压缩、输出上限、工具结果和错误等情况选择不同的继续路径。
- 静默恢复：可自动解决的问题在内部压缩、重试或调整，不把恢复成本直接丢给用户。

产品上，Agent Loop 不只决定“能不能完成”，还决定用户是否觉得快、稳，以及遇到异常时会不会失去信任。

## Prompt System

项目把 Claude Code 的系统提示词概括为 7 层递进架构。提示不是一大段静态文字，而是由基础身份、工具说明、行为规则、风险约束、项目上下文和运行时状态等信息逐层组合。

值得重点理解的设计：

- 反模式接种：直接告诉模型常见错误行为及其后果。
- 负面清单：明确哪些动作不应做，减少模型自己猜边界。
- 爆炸半径：根据动作可能影响的范围调整谨慎程度。
- 内外分层：稳定规则与动态上下文分开管理。
- 按需注入：只加入当前任务真正需要的信息，保护 Token 预算和缓存。

对产品而言，Prompt System 是行为策略层。它需要和工具权限、界面确认及评测一起设计，不能独自承担所有安全责任。

## Tool System

Claude Code 有 66+ 内置工具，第三方 MCP 工具也进入同一套接口和执行流水线。统一协议让注册、参数校验、权限检查、并发、结果返回和错误处理能够复用。

几个关键取舍：

- 只读工具自动并行，写操作自动串行。
- 工具调用在流式响应阶段尽早解析和启动。
- 大于约 100K 字符的工具结果落盘，只把摘要与路径放入上下文。
- 编辑前强制读取，并用唯一性约束、mtime 检查等方式降低错误修改。
- 工具不是越多越好，需要通过延迟发现和按需加载控制上下文成本。

## Context Compression

长会话不会等到彻底超限才一次性摘要，而是逐级处理：

1. 裁剪：先缩短历史中的大型旧工具输出。
2. 去重：移除重复信息，尽量不损失语义。
3. 折叠：收起不活跃片段，同时保留恢复原文的可能。
4. 摘要：最后才让子 Agent 对整体对话生成结构化摘要。

每一级释放出足够空间后就可以停止。压缩完成后，系统还会恢复最近编辑的 5 个文件内容，并重新激活相关 Skills，减少“上下文变短了，但模型忘了正在做什么”的问题。

## Memory

Memory 处理的是跨会话或跨任务仍值得保留的信息。项目分析了 4 类记忆、语义召回、异步预取和后台记忆提取 Agent。

我需要关注的不只是“如何存”，还包括：

- 用 side query 判断哪些记忆与当前任务相关，避免全部塞入上下文。
- 在用户思考或输入期间异步预取，减少等待。
- 用封闭分类和更新规则抑制记忆漂移与重复。
- 区分个人记忆、项目记忆和团队共享记忆的权限边界。

Memory 的产品目标不是让系统记得越多越好，而是让正确的信息在正确时刻出现，并且用户能够理解和控制。

## Skills

Skills 把特定任务的说明、资源与执行方法封装为可复用能力。Claude Code 从 6 层来源发现 Skills，并通过优先级处理覆盖关系。

关键机制包括：

- 懒加载：先暴露简短描述，真正使用时再加载完整内容。
- Token 预算：避免大量 Skills 同时挤占上下文。
- Inline / Fork：简单 Skill 注入当前上下文，复杂 Skill 可交给隔离的子 Agent。
- 权限白名单：Skill 只能使用明确允许的工具和能力。
- 压缩后保留：上下文压缩后重新激活仍相关的 Skill。

Skill 是“如何完成一类任务”的产品化封装，它连接了领域知识、工具和质量标准。

## MCP

MCP 让 Claude Code 动态发现并调用外部工具。项目把它放在工具系统中理解：MCP 工具接入后，应尽量复用内置工具的参数处理、权限、执行和结果回传机制。

值得继续研究的内容：

- 7 种传输方式各适合什么部署环境。
- 连接状态机如何处理发现、断线、重连和能力变化。
- OAuth 2.0 + PKCE 如何完成用户授权。
- 外部工具是否应拥有与内置工具相同的信任等级。
- 工具数量增长后怎样做到按需发现，而不是污染上下文。

## Multi-Agent

项目总结了三种多 Agent 形态：

- 子 Agent：主 Agent 分派一个隔离任务，等待结果返回。
- 协调器：只负责拆分和分配任务，不能自己读写代码，强制保持编排职责。
- Swarm：多个命名 Agent 独立工作，并通过信箱进行点对点通信。

多个 Agent 并行编辑时，Claude Code 用 Git Worktree 为它们提供独立代码副本，降低文件冲突。这里最有价值的产品问题是：怎样让并行分工提升结果，而不是把协调成本和混乱暴露给用户。

## 对产品设计的启发

- **速度是一种系统设计。** 流式输出、预执行和并行启动共同塑造“响应很快”的感受。
- **错误恢复优先于错误展示。** 能安全自动恢复的问题先内部解决，再把真正需要决策的部分交给用户。
- **风险需要纵深防御。** Prompt、规则、AST 分析、确认和 Hooks 各守一层，不能押注单点。
- **渐进处理优于一刀切。** 上下文压缩从低损耗手段开始，只在必要时进入摘要。
- **复杂度应按需出现。** 工具和 Skills 懒加载，让模型与用户都只面对当前相关信息。
- **状态必须可追踪。** 长任务需要明确当前步骤、依赖、阻塞和恢复位置。

## 对 MobileFlow 的启发

- 为移动端操作建立统一 Action 接口，让点击、输入、读取和外部能力共享权限与结果协议。
- 将页面读取等无副作用动作并行，把提交、支付、删除等写操作串行并确认。
- 用流式页面理解和动作预取降低等待，但在执行前保留最终安全闸门。
- 设计渐进式上下文压缩：优先裁剪旧截图和冗长页面数据，保留最近操作目标与关键状态。
- 为长 Flow 设计 Continue Sites，让登录失效、页面变化和网络错误能从局部恢复。
- 多 Agent 探索可以隔离页面理解、流程规划和验证，但要避免多个执行者同时控制同一设备。

## 对 Insight Skills 的启发

- 把研究方法设计成分层 Prompt + Skill，而不是一条越来越长的万能指令。
- 检索、归纳、反例检查和结论表达可以分给不同子 Agent，但必须共享证据引用。
- 大型资料结果应落盘或进入资料库，上下文只保留摘要、来源与按需读取路径。
- 通过语义 Memory 召回用户、行业和项目背景，同时防止旧判断污染新研究。
- 为洞察任务设计 Continue Sites：资料不足时继续检索、证据冲突时回到验证、结论过宽时缩小范围。
- 最终输出应把事实、推断和建议分层，让用户知道哪些内容可靠、哪些仍需判断。
